## Architecture: Core Systems

### i18n System
- Language stored in `localStorage` under key `"language"` — values: `"de"` or `"en"`
- Default language: `"de"`
- `applyTranslations(lang)` in `script.ts` fetches `lang/de.json` or `lang/en.json` via `fetch()`
- `basePath` is auto-detected: `"../lang/"` for subpages (inside `projects/`), `"lang/"` for root
- All translatable DOM elements use `data-i18n="key"` attribute
- After applying translations: `document.body.classList.add("i18n-ready")`
- **Never hardcode visible text in HTML** — always add a `data-i18n` key and a corresponding entry in both `lang/de.json` and `lang/en.json`

### FOUC Prevention
- `css/utilities.css` sets `body { visibility: hidden }` globally
- `script.ts` adds `i18n-ready` class to body after translations load → `body.i18n-ready { visibility: visible }`
- Safety catch-block also adds `i18n-ready` so the page never stays invisible on JS error
- Pages without `script.ts` (e.g. `maintenance.html`) need an inline `<style>body { visibility: visible !important; }</style>`

### Mobile Navigation (Project Subpages)
- Every project page has `<nav class="project-nav sticky-nav">`
- Inside: `<div class="nav-content-wrapper" id="project-nav-links">` and `<div class="mobile-header-controls">` containing lang button + hamburger `id="project-hamburger"`
- **CSS Selector Rule:** ALWAYS use `.project-nav .nav-content-wrapper` — NEVER just `.nav-content-wrapper` globally (causes conflicts with desktop reset)
- Desktop reset: `.project-nav .nav-content-wrapper` is `display: flex; position: static; visibility: visible !important` (outside media query)
- Mobile overlay: `position: fixed; width: 100%; visibility: hidden` → `.active` → `visibility: visible`
- **Never use `width: 100vw`** — use `width: 100%` (100vw causes horizontal scroll due to scrollbar width)
- JS in `script.ts`: toggle `.active` on `projectNavLinks` and `projectHamburger`; outside-click closes menu

### Dynamic Project Rendering
- **Single source of truth for projects:** `src/ts/projects-data.ts`
- `HERO_PROJECT_ID` in `projects-data.ts` controls which project is shown as hero on `index.html`
- All other projects go to the archive automatically
- `project-renderer.ts` injects project cards into `index.html` DOM — never hardcode project cards in HTML
- `Project` interface in `types.d.ts` — all fields must match when adding a project

### Web Components
- `<app-header base-path=".">` — used on `index.html` and top-level pages
- `<app-header base-path="..">` — used on `projects/*.html` subpages
- `<app-footer base-path=".">` / `<app-footer base-path="..">` — same logic
- Scripts loaded at bottom of `<body>`: `js/components/AppHeader.js` + `js/components/AppFooter.js`

### AI Bot System
- `bot.ts` injects the chat widget HTML into `<body>` dynamically
- Reads `localStorage.getItem("language")` to determine DE or EN mode
- Sends `{ query: string, lang: string }` to backend
- Backend URL: `https://portfolio-bach-seven.vercel.app/api/chat`
- `api/index.py` receives `lang`, prepends a mandatory language instruction to the system prompt
- System prompt is loaded from `yusef_brain.md` at runtime
- Failover: `gemini-2.5-flash` → `gemini-2.5-flash-lite` → `gemini-2.0-flash` → `gemini-2.0-flash-lite`
- Client-side graceful fallback: localized error messages in DE or EN, never a raw error
- `GEMINI_API_KEY` is a Vercel environment variable — never in code

### Maintenance Mode
- `status.ts` / `js/status.js` is loaded on every page
- If `MAINTENANCE_MODE = true`, all pages redirect to `maintenance.html`
- `maintenance.html` has inline `body { visibility: visible }` override

|---|---|

## Known Quirks & Gotchas

- `projects-data.ts` is loaded as a script (not a module) — `projectsData` and `HERO_PROJECT_ID` are global variables, not ES module exports.
- `isSubFolder` detection in `script.ts` is based on `window.location.pathname.includes('/projects/')` — all subpages must live in `projects/`.
- The bot greeting message is injected once on `DOMContentLoaded` — it reads `localStorage.getItem("language")` at that moment, so it reflects the language at page load, not at chat-open time. This is intentional and acceptable.
- `maintenance.html` has no `script.js` — it needs the inline visibility override or the page is invisible.
- The `archive.html` page renders all non-hero projects dynamically — it also loads `project-renderer.ts` but with a different render target.
- `tsconfig.json` uses `strict: false` but `noImplicitAny: true` — explicit `any` is allowed, implicit is not.
- EmailJS public key is client-visible — the domain `yusefbach.de` must be whitelisted in the EmailJS dashboard to prevent abuse.
